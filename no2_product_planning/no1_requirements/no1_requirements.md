# Requirements

## 記帳習慣很難養成

- **問題根源：**
    - 角度：使用者
    - 記帳本身是一件很瑣碎的事
    - 因為瑣碎所以很容易忽略
    - 忽略了一筆兩筆三筆，帳本就會壞掉
    - 帳本壞掉又很難想起來前面花過哪些錢
    - 有心的話會重新開始，但通常是直接放棄
- **評估方案:**
    - OS AI 紀錄
      - **描述:**
        - 透過 Device OS 層 AI 自動爬取 mail, 簡訊, 支付 App 通知，自動記錄收支紀錄並記錄於待審區
        - 使用者開啟 App 後 review 待審區，確認無誤後逐一或一鍵確認
      - **效益:**
        - 降低記帳門檻
        - 提高記帳準確度
        - 提高記帳資訊濃度，可供營運人員更精準的消費歷程與細節
      - **風險:**
        - 隱私問題
        - 準確度問題
        - 耗電問題
        - 開發難度極高
    - iOS Widget
      - **描述:**
        - iOS Widget 可提供
          - 快速記帳介面
          - 開啟收入/支出記錄頁面快速入口
      - **效益:**
        - 降低記帳門檻
      - **風險:**
        - 無
    - App 開啟預設扣帳畫面
      - **描述:**
        - App 可設定開啟預設畫面為扣帳畫面
      - **效益:**
        - 降低記帳門檻
      - **風險:**
        - 不一定每次開記帳 App 的動機都是扣帳

## 記帳報表無法應對真實生活的複雜度

- **問題根源：**
    - 角度：使用者
    - 記帳裡的帳戶不一定等於真實世界裡的銀行帳戶
    - 記帳裡的帳戶可能有夫妻共同花費帳戶以及個人花費帳戶
    - 當要找出個人當月花費的時候，可能因為夫妻協議共同花費7:3分擔支出，導致算帳時需要花費複雜的操作來查看花費
    - 不管是記帳 App 或是試算表，當遇到這樣的情形的時候都會需要相當程度的操作
- **評估方案:**
    - Web Console 報表，可自訂維度與計算式
      - **描述:**
        - 提供類似 JIRA JQL 功能，使用者自定義報表計算式
        - 提供計算式記憶功能，省去使用者需要重複輸入計算式
      - **效益:**
        - 解決複雜報表需求
      - **風險:**
        - DDOS 攻擊
        - 額外後端伺服器成本

## 產品收費串接 IAP

- **問題根源：**
    - 角度：營運
    - 做功能給使用者，使用者付費
- **評估方案:**
    - 原生 IAP Apple StoreKit 與 Google Play Billing
      - **描述:**
        - 直接串接官方 IAP API 進行收費
      - **效益:**
        - 無第三方中介層 RevenueCat 額外抽成
        - 無第三方服務的依賴風險
      - **風險:**
        - 需要自行處理跨平台 Google Play Console 與 App Store 的實作差異與收據驗證
    - RevenueCat 已排除
      - **描述:**
        - 使用 RevenueCat 進行收費
      - **排除理由:**
        - 引進第三方中介並產生額外平台費，考量長期營運成本與去依賴化，改為自建 IAP 封裝層

## 雲端同步架構

- **問題根源：**
    - 角度：工程
    - 付費版需支援跨裝置資料備份與同步
    - 需在開發速度、成本可控、離線優先三者之間取得平衡
- **評估方案:**
    - Google Firestore + 增量批次同步
      - **描述:**
        - 使用 Google Firestore 作為付費版跨裝置備份與同步後端
        - 搭配 Firebase Auth 管理使用者身份與權限
        - 採增量批次同步模型：本地資料優先寫入，上傳至 Firestore，非即時同步
        - 免費版資料儲存於本機；付費版啟用批次同步
      - **效益:**
        - 託管服，無需自行維護伺服器、資料庫與 API
        - 與 Firebase Auth 如 Google 登入無縫整合
      - **風險:**
        - 成本與讀/寫次數掛鉤，非固定月費，需謹慎設計同步邏輯
        - 未來遷移至自建後端的成本較高

## 同步配額管理

- **問題根源：**
    - 角度：工程
    - Firestore 成本與讀寫次數掛鉤，若因程式 Bug 如無窮迴圈或設計不良，可能導致單一裝置 Quota 暴增
- **評估方案:**
    - 本地計數 Local Storage
      - **描述:**
        - 採用本地儲存計數器追蹤同步操作次數，不依賴 Firestore 進行計數查詢
      - **效益:**
        - 避免雲端計數在每次同步時額外消耗讀寫 Quota，即 Quota Tax
      - **風險:**
        - 僅能限制單一裝置，無法限制使用者多裝置的總量

## 認證方案

- **問題根源：**
    - 角度：工程
    - App 需要綁定帳號以支援資料雲端備份與多裝置同步
    - 必須在訪客可直接使用的 Local-First 模型，與強制登入牆的 Auth-First 模型兩種存取策略中做出決策
- **評估方案:**
    - App 存取模型: Local-First，情境式登入
      - **描述:**
        - 採用情境式登入，登入不為使用 App 的前提，使用者可作為訪客離線記事，僅在啟用付費功能或雲端備份時被要求綁定帳號
      - **效益:**
        - 極低的起始使用門檻
      - **風險:**
        - 訪客升級正式帳號時，需要處理本地資料是否合併或覆蓋的邊角問題
        - 增加 UI 導航的複雜度，需同時處理未登入狀態與登入狀態的主畫面切換
    - App 存取模型: Auth-First，強制登入牆 ★ 選用
      - **描述:**
        - 強制依賴 Firebase Authentication 作為 App 的獨立存取門檻，未認證者無法進入主頁
        - 首次登入後，Firebase Auth 底層快取憑證，確保 App 可持續支援斷網離線運作
      - **效益:**
        - 資料強制歸戶明確，消除匿名資料遺失的客服風險
        - 降低後續帳號合併與權限管理的系統複雜度
      - **風險:**
        - 會流失排斥登入流程的潛在新用戶
    - 提供身份: 僅 Google 登入 ★ 選用
      - **描述:**
        - 只支援 Google Sign-In，排除 Apple 登入等其他身分
      - **效益:**
        - 與 Firestore 的權限管理可無縫整合
      - **風險:**
        - iOS 使用者可能更偏好 Apple ID，造成使用摩擦

## 多幣別報表架構

- **問題根源：**
    - 角度：工程
    - 使用者擁有多幣別帳戶，報表彙總時需要統一的計算基準，否則跨幣別金額無法相加比較
- **評估方案:**
    - 使用者設定基礎貨幣，統一換算
      - **描述:**
        - 使用者可設定基礎貨幣
        - 所有非基礎貨幣的資產，在顯示於多幣別報表時，必須換算為基礎貨幣價值
      - **效益:**
        - 報表具有統一的計算基準，可進行跨幣別彙總與比較
      - **風險:**
        - 換算結果依賴使用者維護的匯率精度

## 匯率架構

- **問題根源：**
    - 角度：工程
    - App 支援多幣別，需決定如何取得與管理匯率
- **評估方案:**
    - 手動維護 + Strict Direct Rate
      - **描述:**
        - 匯率不使用外部 API，由使用者手動輸入，或由跨幣別轉帳時自動記錄
        - 系統僅使用外幣到基礎貨幣的嚴格直接匯率 Strict Direct Rate 進行報表計算
        - 若無直接匯率，視為無法計算，不進行中間跳轉
        - 報表換算時不回溯資產當時匯率，使用現行匯率
      - **效益:**
        - 確保 App 100% 離線運作，不依賴外部服務
        - 避免即時匯率 API 費用，且費用難以預估
        - 換匯交易時，App 不限制使用者輸入的匯率，可反映私下換匯情形
      - **風險:**
        - 匯率精度依賴使用者自行維護
        - 若無外幣到基礎貨幣的直接匯率，相關帳戶的報表數據無法計算

## 報表時區設計

- **問題根源：**
    - 角度：工程
    - 使用者可能跨時區使用 App，今天的定義需與使用者時區一致
    - 若不處理時區，報表的時間邊界如當日起訖，會不符合使用者的真實認知
- **評估方案:**
    - 使用者設定資料顯示依據時區，所有報表以此為基準
      - **描述:**
        - 所有報表的時間邊界必須基於使用者設定的時區進行計算
      - **效益:**
        - 確保報表中今天的定義符合使用者所在時區
      - **風險:**
        - 無

## 本機資料庫

- **問題根源：**
    - 角度：工程
    - App 需要高效率的本機結構化資料儲存，支援複雜查詢、離線優先架構，以及付費版的雲端同步
    - 需在原生 SQLite、AsyncStorage、WatermelonDB 之間取捨
- **評估方案:**
    - WatermelonDB ★ 選用
      - **描述:**
        - 基於 SQLite 的 React Native 本機資料庫，提供響應式查詢與 Lazy Loading
        - 官方支援 Sync Adapter 介面，可對接 Firestore 等後端做增量同步
      - **效益:**
        - Lazy Loading 確保大量資料下 UI 不阻塞
        - Sync Adapter 與批次同步架構天然相容
      - **風險:**
        - 需學習 WatermelonDB 的 Schema 定義與 Model 層設計
    - 原生 SQLite
      - **排除理由:**
        - 無 ORM、查詢需手寫 SQL，維護成本高；缺乏官方 Sync 支援

## 定期交易機制

- **問題根源：**
    - 角度：工程
    - 使用者可設定定期交易如每月房租
- **評估方案:**
    - 本機觸發 + Offline Loophole Tolerance
      - **描述:**
        - App 啟動時依據本地資料執行定期交易檢查，不強制驗證雲端權限
        - App 啟動時必須補產生從上次檢查到今天所有遺漏的交易
      - **效益:**
        - 確保使用者在無網路環境下，定期交易仍能正常產生，不中斷記帳體驗
      - **風險:**
        - 若定義為付費功能，訂閱過期但尚未連網同步的使用者，仍可暫時獲得定期交易產生的新紀錄，屬權限漏洞
