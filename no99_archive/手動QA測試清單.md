    - 範圍為 impl 全模組
- 手動 QA 人員照此逐項驗收
- 涵蓋 `no2_accounting_app` 前端
- 涵蓋 `no3_cloud_functions` 後端
- 產生日期：2026-07-02
- 來源：impl 全模組程式碼與 spec 掃描
    - 九模組平行調查 + 完整性稽核

## 如何使用這份清單

- 分四輪跑，由輕到重
- 煙霧測試輪
    - 只跑 `P0`
    - 每次改版必跑

- 模組深測輪
    - `P1` `P2` 全跑

- 裝置矩陣輪

- 優先級
    - `P0` 壞了不能出貨
    - `P1` 重要
    - `P2` 次要
- 類型

- ⚑ 以下行為 spec 未定義、以 impl 現況為驗收基準：計算機運算語意

#### ⚠ 此模組測試加壓點

- 金額無上限保護
    - ×10000 整數儲存，約 9 千億以上超過 JS 安全整數
    - 超大值會精度失真，存讀一致性必壓
- 計算機隱藏狀態
    - 未按 = 的暫掛算式在儲存時自動結算
    - 負數結果被靜默取絕對值存入
    - 使用者所見與實際儲存值可能不同
    - 此區 spec 未定義、只有 impl 行為
- 週期回補邊界
    - `MONTHLY`/`YEARLY` 31 號錨定
    - `endOn` 當日含入，day-granular
    - 離線多日再開 app 的回補
    - 重複產生或漏產都是高風險
    - 並行觸發依賴 in-flight 去重
        - 並行情境如登入 + 建排程同時
- 此筆及未來的資料切割鏈長
    - 舊排程 `endOn` 回推一週期 → 軟刪既有實例 → 重建新排程回補
    - 任一步錯就出現孤兒或重複實例
    - 對應 Undo 要同時還原 `endOn` 與多筆實例
    - 需重點驗證
- K-mode 與幣別小數位三層換算
    - K-mode 即千元單位
    - 三層換算：輸入 → 儲存 → 顯示
    - 切換不同幣別帳戶時 `decimals` 即時變動
    - 已輸入的超位小數是否殘留需驗
- 轉帳幣別非同步載入
    - 帳戶幣別查詢是 async
    - 快速切換帳戶時可能短暫錯態
        - 同幣別/跨幣別判定
        - `to` 金額 disable 狀態
- 額度卡點只在新增路徑檢查
    - 新增路徑即 `createTransaction`/`createTransfer`
    - 編輯與排程回補不卡
    - 降級後舊排程持續產生實例
    - 是否符合商業預期需確認
- 轉排程順序 spec 與 impl 不一致
    - spec 條列先刪後建
    - impl 先建後刪防資料遺失
    - QA 驗收失敗情境時以 impl 為準
    - 建排程失敗原紀錄必須完好

#### ⚠ 此模組測試加壓點

- PeriodPage 的錯誤重試按鈕
    - 只呼叫 `setError(null)`
    - useEffect 依賴未變可能不重新 fetch
    - 畫面恐停在 Loading
    - 錯誤路徑要實測重試是否真的重載
- 期間切換 chevron
    - 在版面上存在但為純 View，停用色
    - 點按無反應
    - spec 佈局列為按鈕但互動未定義
    - 易被回報成按鈕壞掉
    - 驗收時先對齊這是否為預期
- 期間邊界依 timeZone 計算
    - 走 `getPeriodDates` 與 `getStartOfDay`
    - 邊界重災區
        - 跨日 23:59 前後
        - 改時區設定後
        - 週/年粒度的起迄日歸屬
    - 需壓日/週/月/年四種粒度
- 單一 slice 滿圈的 donut 幾何
    - 即僅支出或僅收入、單分組
    - 曾有 arcToOval ±360° 退化破圖史
    - sweep 動畫結尾 frame 與極端比例 99.9%:0.1% 要視覺盯
- 快取一致性鏈長
    - 多條失效路徑
        - `useFocusEffect` 重載
        - `undoRevertCount`
        - `ratesVersion`
        - `clearCache`
    - 任何遺漏會讓清單顯示舊資料
    - 改匯率、改主要貨幣、Undo 復原後，都要驗清單即時性
- 新增未來期間紀錄改變 future ceiling
    - 觸發 FlatList remount
    - 滑動位置會重置回當前期間
    - 連續操作下注意頁面跳動與 `activeIndex` 錯位
- 換帳號瞬態守衛
    - 登出換帳號後短暫窗口，`selectedAccountIds` 仍是前帳號的
    - 實作跳過 fetch 避免報表瞬空
    - 換帳號流程要驗首頁不閃空白，也不顯示他人資料
- 清單金額一律顯示絕對值
    - 靠分組與轉帳箭頭表達方向，分組即支出/收入焦點
    - 金額本身無正負號與紅綠色
    - 測試預期不以顏色或符號判斷收支
- 單幣別/多幣別合計基準不同
    - 原幣 vs 換算主要貨幣
    - 切換帳戶選取改變幣別組成時
        - focus 卡合計與分組小計，數值基準會整組切換
        - 容易誤判為算錯

#### ⚠ 此模組測試加壓點

- 刪除級聯範圍
    - 轉帳以 from 或 to 任一端命中就一併軟刪
    - QA 必須雙向都驗
        - A→B 與 B→A 都要在刪 A 時消失
    - 漏測轉入端會放過半套級聯的 bug
- Undo 用刪除當下的 id 快照還原
    - 不重查
    - 復原列消失後帳戶與其交易永久軟刪
    - app 內無其他復原入口
    - QA 要驗 undo 時效內外兩種路徑
- 名稱完全沒有查重、長度上限與 trim
    - 僅擋全空白
    - 重複名稱在選擇器與合併畫面，可能造成辨識混淆
    - 前後空白會原樣入庫
- 配額邊界不對稱
    - `createAccount` 是達到 3 就擋
    - 記帳/轉帳是超過才擋，匯入可能推超額
    - QA 壓邊界值 =3 時各動作的放行差異
- 帳戶選擇器 value 失效的靜默改選
    - `resolveAutoPick` 會靜默自動改選第一個可用帳戶
    - 編輯舊轉帳而其帳戶已被停用時，from/to 會被無提示換掉
    - 屬資料正確性高風險
- 停用與刪除是兩套語意
    - `disabledOn` vs `deletedOn`
    - 停用仍佔配額、仍在列表
    - QA 容易混淆兩者對選擇器、Home 篩選、配額三處的不同影響
- 編輯模式幣別鎖定是雙保險
    - UI disabled + 資料層不寫 `currencyCode`
    - 但佔位匯率只在建立時種一次
    - 若未來放開改幣別，既有交易金額與匯率會對不上
- 帳戶本身無餘額欄位
    - 刪帳戶對數字的影響，只反映在 Home 期間收支
    - QA 驗刪除後應以首頁數字變動為準
    - 別找不存在的帳戶餘額畫面
- 拖拉排序與 ScrollView 手勢共存
    - 拖拉時暫停捲動
    - 長列表拖到畫面邊緣的行為值得加壓
    - 排序寫入失敗值得加壓
        - 僅 `console.error`、無使用者提示

    - 合併前已刪除的轉帳不進快照

    - 若合併後 4 秒內使用者又動了資料

- 全域單一 Undo 槽位
    - 合併後任何一次刪除操作 → 覆蓋合併的復原機會
    - 使用者可能以為還救得回來
- 相容性防護有兩道
    - 相容性指帳戶同幣別／類別同型別
    - 兩道為 UI 預過濾＋存檔前防呆
    - 正常流程觸不到錯誤路徑

- 合併在單一 `database.batch` 內執行
    - 中途失敗理論上整批回滾
    - 並彈 `合併失敗`
    - 手動難以觸發

#### ⚠ 此模組測試加壓點
- 付費牆 impl 與 spec no26 有落差
    - spec 載明 `暫不升級` 連結與自動續訂揭露文案區
        - 揭露內容含訂閱週期、款項向 Apple ID 收取
    - impl 只有 header X 與 `Cancel anytime` 一句
    - 恢復購買 spec 只分成功/失敗，impl 多了空結果中性提示
    - 上架審查與驗收基準要先裁決以哪版為準
        - 上架審查指 App Store 自動續訂揭露要求
- 跨帳號綁定拒絕在 client 無錯誤呈現
    - 拒絕即後端回 already-exists
    - `restorePurchases` 查到 StoreKit 購買就跳 `Purchases restored`
    - 但後端拒絕歸戶、等級不升
    - 成功訊息與實際等級矛盾，易被誤判為 `恢復壞了`
- 購買成功關頁依賴 server 驗證 + Firestore 推送整條鏈
    - 弱網下只有 90 秒 timeout 防呆，期間整頁鎖操作
    - QA 要在弱網/斷網切換下加壓
    - 確認不會卡死或雙重提示
- 配額邊界不對稱
    - 帳戶/分類是達上限就擋新增，即 `>= 3` 與 `>= 7`
    - 記帳是超額才擋，即 `> 3` 與 `> 7`
    - 恰在邊界的值最容易出錯
    - 只有匯入能造成超額，測試要靠匯入預置資料
- 換帳號選 Clear 是不可復原硬重置
    - 雲端不自動還原，備份為 up-only
    - 誤按即資料全失
    - 對話框文案與 destructive 樣式要驗
    - 確認 Clear 只清本機、前帳號雲端資料完好
- 離線授權以 AsyncStorage 快取判定
    - `expirationDate` 為 `null` 視為無期限
    - 若 server 對訂閱型商品誤寫 `expires_date` 為 `null` → 付費者離線將永久有效
    - sandbox 訂閱時間加速，時序類測試容易誤判
        - 時序類指到期、寬限期
    - 需比對 Firestore `entitlements/{uid}` 的 `status` 欄位
- 退款通知早於 client 驗證的重送鏈
    - webhook 回 503，靠 Apple 重送
    - 若 client 始終未完成 `verifyTransaction` 歸戶
    - 退款事件可能在 Apple 放棄重送後永久遺失
    - 需搭配後端 log 驗證重送鏈
- TestFlight/debug 版的 Mock Tier 會覆寫顯示等級
    - 覆寫 `currentTier` 但不影響 `serverTier`
    - 測購買/恢復時若忘記關 mock → 畫面等級與真實授權不一致
    - 易汙染測試結論
    - production 版須驗證 Mock 入口完全不存在

#### ⚠ 此模組測試加壓點
- 舊版升級 migration 鏈是最脆弱點
    - v2、v6 為空 step 佔位，維持版本連續
    - 任何 schema 改動漏補 migration → 舊裝置覆蓋安裝直接閃退
    - QA 必須實測舊版建資料 → 覆蓋安裝的路徑
    - 不能只測全新安裝
- 換帳號選清除是整庫硬重置
    - 即 `unsafeResetDatabase`
    - 雲端備份為單向上傳、永不下載還原
    - 使用者誤按等於資料永久消失
    - 對話框文案、destructive 樣式與不可略過性要逐語系重驗
- 多帳號資料同庫共存
    - 隔離完全依賴每條查詢帶 `user_id` 過濾
    - 任何新增畫面或查詢漏帶 → 跨帳號帳務外洩
    - QA 應在雙帳號共存裝置上驗
    - 每個列表與搜尋都跑過
- undo 僅單步、4 秒且不持久化
    - 倒數起算依賴路由判斷
        - 離開觸發編輯器才起算
    - 非典型導航可能吃掉復原窗口
        - 如刪除後直接跳往其他頁
    - 復原前殺 app → 刪除直接定案
- 時區同時參與期間邊界計算與報表快取 key
    - 跨日交易切時區後的分頁歸屬、統計重算要人工核對
    - 容易出現差一天
- 同步引擎全部行為在 UI 上不可見
    - 含冷卻、per-user watermark、離線 skip、失敗 30 秒退避、批次 500 筆
    - QA 驗證只能靠 debug log 或 Firestore console
    - 容易漏測
- ErrorBoundary 的 Retry 只重置 `hasError` 狀態
    - 錯誤源仍在 → 立刻再進錯誤畫面
    - 觀感上像按鈕無效
    - 錯誤畫面為自含樣式，不跟主題深淺色
